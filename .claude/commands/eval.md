小テスト結果と生徒メモのCSVをもとに、実施した単元を考察し、単元の `02_評価まとめ.md` に追記する。

## 入力情報の受け取り方

引数（$ARGUMENTS）がある場合はそれを使う。なければユーザーに以下を確認する：

- 年度・科目（例：2026 / 情報Ⅰ）
- 実施した単元（番号・単元名。複数可。実施途中の単元はその旨）
- CSVファイル（添付または保存先のパス）
  - `quiz_export_*.csv`（小テストの設問別正答率・誤答内訳・記述回答サンプル）
  - `memo_export_*.csv`（授業別の生徒メモ）
- 次回の単元（任意。考察の参考にする）

### CSVの読み込みについて（重要）

- CSVは**実ファイルを直接読む**こと。会話に貼り付けられた内容は文字化け（UTF-8をLatin-1として表示）していることがあるため、添付の元ファイル（例：`~/Downloads/` 配下）を Bash の `cat` 等で読み直す。
- `quiz_export` の主な列：`レッスン, Q番号, 問題文（要約）, 形式, 全体, （組ごとの正答率）`。下部に「誤答内訳（正答率60%以下）」「記述回答サンプル」がある。
- `memo_export` は `レッスン, メモ内容（結合・要約）`。

## 考察の前提・注意（必ず守る）

- **参加率は考察しない**：小テスト・メモの実施は生徒の任意。解答率・メモ率の多寡には言及しない。
- **少人数のノイズに注意**：対象人数（ヘッダの「対象: N名」）を確認する。人数が少ないほど正答率は1人で大きく動く（例：11名なら1人で約9〜18%）。断定を避け、組別の0%/100%は「内容の弱点」ではなく**特定の個人**の可能性として扱う。
- **補習・演習の前後を確認**：低い正答率が授業内の補習・計算演習などの**前**の数値である場合、「定着が確認できた弱点」と断定せず「重点的に補いたい領域」として扱う。判断がつかなければユーザーに確認する。
- **評価列は埋めない**：観点別評価結果テーブルの「評価」列（評定）は教師が記入する欄。スキルでは「所見」のみ埋める。

## 分析の進め方

1. `quiz_export` の `全体` 正答率で、定着している内容（高い）とつまずいている内容（低い）を仕分ける。
2. `形式` に注目する。`並び替え` の低さは**手順・順序づけ**の弱さ、`記述式` の回答サンプルは**思考・判断・表現**の質を示す。
3. 「誤答内訳」で誤概念を特定する（どの誤答に集中しているか。例：1024と1000の混同、線形と指数の混同など）。
4. 「記述回答サンプル」で記述の質・言語化の強みを評価する。
5. `memo_export` の内容から、関心の強いテーマ・理解の深さを読み取る。
6. 単元の `01_単元計画.md` を Read し、単元目標・授業構成・時間配分と照らして達成度を評価する。

## 作業手順

1. 作業ブランチを作成・切り替える：
   ```bash
   git checkout -b fix/YYYY-informaX-NN-eval
   ```
2. 対象単元の `docs/YYYY/informaX/NN_単元名/01_単元計画.md` を Read する。
3. 上記「分析の進め方」に沿って考察する。
4. `docs/YYYY/informaX/NN_単元名/02_評価まとめ.md` を Read し、後述のフォーマットに沿って追記する。
5. 追記内容を報告してレビューを促す。
6. ユーザーがレビューOKを出したら GitHub にプッシュする。

## 02_評価まとめ.md への書き込みフォーマット

テンプレートの構成（`# 単元名 評価まとめ` / `## 実施期間` / `## 観点別評価結果` / `## 気づき・課題`）を維持したうえで、以下を埋める。

### 冒頭の前提注記（必要な場合）

少人数・実施途中・補習前の数値など、解釈の前提があれば本文の先頭に引用（`>`）で明記する。例：

```markdown
> **前提**：対象は◯名。正答率は1人で約X%変動するため、組別の0%/100%は特定の個人のつまずきを反映している可能性がある。
```

### 観点別評価結果（所見のみ記入）

「評価」列は空のまま、「所見」列にデータに基づく所見を記入する。

```markdown
| 観点 | 評価 | 所見 |
|---|---|---|
| 知識・技能 |  | （高定着の内容・つまずいた内容を正答率の根拠とともに） |
| 思考・判断・表現 |  | （記述の質・並び替え/手順の傾向） |
| 主体的に学習に取り組む態度 |  | （メモから読み取れる関心・取り組み姿勢） |
```

### 気づき・課題（考察の本体）

`## 気づき・課題` の配下に、以下の小見出し（`###`）で考察を記述する。実施途中の場合は冒頭に「※実施範囲・暫定」の注記を付す。

- `### 目標の達成度`
- `### 授業構成・時間配分`
- `### ストロングポイント（得意）`
- `### ウィークポイント（つまずき）`
- `### 興味関心`

### 次年度に向けた妥当性の考察（H2セクションを追加）

`## 気づき・課題` の後に、次年度向けの2セクションを追加する。

```markdown
## 扱った内容の妥当性（次年度に向けて）

- **全体評価**：（構成・題材が妥当か）
- **修正した方がよい部分**：（踏み込みすぎ・分かりにくい箇所など）
- **追加した方がよい内容**：
- **削除候補**：

## 小テストの妥当性（次年度に向けて）

- **難易度**：（易しすぎ／難しすぎ・識別力の有無。少人数ならその前提を明記）
- **機能している設問／見直したい設問**：
- **追加した方がよい問題**：
- **削除・見直し候補**：
```

各項目は、データ（正答率・誤答内訳・記述サンプル・メモ）に基づく具体的な根拠とともに記述する。該当がない場合は「明確に不要な内容はない」等を明記する。

## 完了後・GitHub へのプッシュ

1. 追記したファイルのパスと内容を報告し、レビューを促す。
2. ユーザーが「OK」「問題ない」などレビュー完了を伝えたら、以下を実行する：
   ```bash
   git add docs/YYYY/informaX/NN_単元名/02_評価まとめ.md
   git commit -m "Add: NN_単元名 評価まとめに考察を追記（YYYY 情報X）"
   git push origin fix/YYYY-informaX-NN-eval
   ```
3. プッシュ後、main へのマージを案内する：
   ```bash
   git checkout main
   git merge fix/YYYY-informaX-NN-eval
   git push origin main
   git branch -d fix/YYYY-informaX-NN-eval
   ```
4. マージ完了後、GitHub Pages のデプロイが自動で走ることを伝える（`.github/workflows/deploy.yml` による）。
