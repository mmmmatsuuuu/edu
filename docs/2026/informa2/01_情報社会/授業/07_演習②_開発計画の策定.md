# 第7時：演習②「開発計画の策定」

## 本時の目標

- デジタルスタンプラリーの実現に向けた問題解決プロセスを整理できる
- 要件定義・スケジュール・役割分担を含む開発計画を立案できる

## キーワード

| 用語 | 説明 |
|---|---|
| 問題解決プロセス | 課題の発見・分析・解決策の立案・実施・評価という一連の思考の流れ |
| 要件定義 | システムやサービスを作る前に「何を実現すべきか」を明確にまとめる作業 |
| 企画書 | 目的・対象・内容・スケジュール・役割分担などをまとめた計画の文書 |
| フィジビリティ | 計画が実際に実行可能かどうかを見極める考え方（実現可能性） |

## 準備物・教材

【教師】
- 演習説明スライド（問題解決プロセスの流れ・企画書テンプレートの説明）
- 開発計画書テンプレート（Google Classroom 配布、Google Docs）

【生徒】
- 学習用端末
- 開発計画書テンプレート
- 前時（演習①）の著作権確認シート（素材リストとして参照）

## 開発計画書テンプレートの構成

```
【プロジェクト名】：デジタルスタンプラリー

■ 目的・背景
（このプロジェクトで何を実現したいか、なぜやるのか）

■ 対象ユーザー
（誰が使うか：来場者・生徒・保護者等）

■ 実現したい機能（要件定義）
□ スタンプ付与機能：
□ ガチャ機能：
□ コンテンツ配信機能：
□ その他：

■ 使用する素材・コンテンツ
（演習①で確認した素材リストから選択し、著作権上の扱いも記載）

■ スケジュール（案）
| 時期 | 単元 | 主な作業 |
|---|---|---|
| 6〜7月 | コンテンツ単元 | 景品コンテンツの制作 |
| 9〜10月 | 情報システム単元 | システムの設計・構築 |
| 11月 | 文化祭 | 本番運用 |
| 12〜2月 | データサイエンス単元 | 運用データの分析 |

■ フィジビリティの確認
（実現が難しいと思う点・解決策の案）

■ 役割分担案
（コンテンツ担当・システム担当・データ分析担当等）
```

## 授業展開

| 時間 | 段階 | 学習活動 | 指導上の留意点 |
|---|---|---|---|
| 10分 | 導入・説明 | 問題解決プロセスの流れを説明。開発計画書テンプレートの各項目の意味と記入のポイントを実演 | 「計画は変わっても構わない。大切なのは今の時点で考え抜くこと」と伝え、完璧な計画を求めるプレッシャーを減らす |
| 30分 | 作業 | グループで開発計画書を作成。①目的・対象ユーザーを言語化②実現したい機能を要件として列挙③フィジビリティの確認（難しい点・解決の方向性）④スケジュールの確認⑤役割分担案の検討 | 演習①の素材リストを参照しながら「何を使えるか」も合わせて検討させる。機能リストは「あったらいいな」ではなく「必ずやる」「できればやる」を区別させる |
| 10分 | 発表・まとめ | グループごとに計画書の要点（実現したい機能・フィジビリティ上の課題）を1〜2分で共有。単元全体の振り返りフォームに記入 | 他グループの発表から「自分たちが見落としていた視点」を拾わせる。単元の学びがプロジェクトの法的基盤・計画立案につながったことを確認する |

## 本時の評価

- **思考・判断・表現**：開発計画書の要件定義・フィジビリティの記述から、問題解決プロセスに基づいた計画立案ができているか評価
- **主体的に学習に取り組む態度**：グループ内での積極的な議論・発表への参加

## 対応する学習指導要領の内容

- 情報Ⅱ（1）ア：情報技術の発展と社会の変化（情報技術を活用した問題解決）
- 情報Ⅱ（1）イ：情報に関する法規・倫理（著作権・知的財産の実務的活用）
