# 第6時：演習①「開発計画の策定（前半）」

## 本時の目標

- デジタルスタンプラリーの目的・対象ユーザーを言語化できる
- 問題解決プロセスの流れを理解し、要件定義の考え方を説明できる
- プロジェクトで実現したい機能を要件として整理できる

## キーワード

| 用語 | 説明 |
|---|---|
| 問題解決プロセス | 課題の発見・分析・解決策の立案・実施・評価という一連の思考の流れ |
| 要件定義 | システムやサービスを作る前に「何を実現すべきか」を明確にまとめる作業 |
| フィジビリティ | 計画が実際に実行可能かどうかを見極める考え方（実現可能性） |
| 企画書 | 目的・対象・内容・スケジュール・役割分担などをまとめた計画の文書 |

## 準備物・教材

【教師】
- 演習説明スライド（問題解決プロセスの流れ・開発計画書テンプレートの説明）
- 開発計画書テンプレート（Google Classroom 配布、Google Docs）

【生徒】
- 学習用端末
- 開発計画書テンプレート
- 第1〜5時のノート・ワークシート（参照用）

## 開発計画書テンプレートの構成（前半で記入する範囲）

```
【プロジェクト名】：デジタルスタンプラリー

■ 目的・背景
（このプロジェクトで何を実現したいか、なぜやるのか）

■ 対象ユーザー
（誰が使うか：来場者・生徒・保護者等）

■ 実現したい機能（要件定義）
□ スタンプ付与機能：
□ ガチャ機能：
□ コンテンツ配信機能：
□ その他：
　※「必ずやる機能」と「できればやる機能」を区別して記入する
```

## 授業展開

| 時間 | 段階 | 学習活動 | 指導上の留意点 |
|---|---|---|---|
| 10分 | 導入・説明 | 問題解決プロセスの流れを確認する。開発計画書テンプレートの全体像と、今日記入する範囲（目的・対象ユーザー・要件定義）を説明する | 「計画は変わっても構わない。大切なのは今の時点で考え抜くこと」と伝え、完璧な計画を求めるプレッシャーを減らす。前半・後半の2回に分けて進めることを説明する |
| 35分 | 作業 | グループで開発計画書（前半）を作成する。①目的・背景を言語化する（「誰のために・何のために」を起点に）②対象ユーザーを具体的に整理する（年齢層・使用環境・ITリテラシー等）③実現したい機能を要件として列挙し、「必ずやる」「できればやる」に区別する | 目的の言語化でつまずいているグループには「デジタルスタンプラリーがなかったらどんな不便があるか」を問いかける。要件は「機能名だけ」でなく「誰が・何をできるか」まで書かせる |
| 5分 | まとめ | 各グループが「要件定義で出た機能」を1〜2個口頭で共有する。次時（後半）でやること（フィジビリティ・スケジュール・役割分担・発表）を確認する | 前半で記入した内容を保存・提出させる。次時は「実現可能性の検討」から始めることを予告する |

## 板書・スライド構成

1. 問題解決プロセスの流れ（発見→分析→立案→実施→評価）
2. 開発計画書テンプレートの全項目一覧と今日の作業範囲
3. 「必ずやる機能」と「できればやる機能」の区別の考え方
4. 次時の予定（フィジビリティ・スケジュール・役割分担・発表）

## 本時の評価

- **思考・判断・表現**：開発計画書の目的・要件定義の記述から、プロジェクトの目的を言語化し必要な機能を整理できているか評価（次時終了後にまとめて提出）

## 対応する学習指導要領の内容

- 情報Ⅱ（1）ア：情報技術の発展と社会の変化（情報技術を活用した問題解決）
