# アプリ設計：デジタルスタンプラリー

本ドキュメントは、03_情報システム単元で構築するアプリ「デジタルスタンプラリー」の**完成像（設計）**を定める。各授業（特に座学06「全体設計を読む」、演習07〜11）と、生徒の担当パーツの粒度は、この設計に紐づく。

## 1. アプリ概要

- **目的**：文化祭の各ブース（出し物）を巡ってもらう回遊施策。来場者がブースのQRコードを読むとスタンプがたまり、規定数に達するとガチャを回してデジタル／実物の景品を獲得できる。
- **利用シーン**：来場者が**自分のスマートフォン**でQRを読み取り、ブラウザで動くWebアプリにアクセスする。専用アプリのインストールは不要。
- **規模感**：来場者数百名程度／ブース数十程度を想定。少人数（生徒約11名）で運用できる軽量さを優先する。

## 2. システム構成

技術構成は **Google Apps Script（GAS）の Webアプリ ＋ Googleスプレッドシート**。

```
[来場者のスマホ]
   │ QR読取（ブースID付きURL）
   ▼
[GAS Webアプリ]  ── doGet/doPost、HTML Service で画面を返す
   │ 読み書き
   ▼
[Googleスプレッドシート]  ── データベースの代わり（複数シート）
```

- **クライアント**：来場者のブラウザ（HTML/CSS/JS）。座学02「クライアント・サーバ」と接続。
- **サーバ**：GAS Webアプリ（`doGet`でアクセスを受け、スプレッドシートを読み書きして結果を返す）。座学05「API」と接続。
- **データストア**：スプレッドシート。座学03「データの管理」と接続。

> GASを使う理由：サーバ構築不要・無料・スプレッドシートと直結で、少人数運用でも扱いやすい。生徒が触る範囲を関数単位に切り出しやすい。

## 3. 利用フロー（来場者視点）

1. 最初にどれかのブースQRを読む、または入口の案内QRを読む。
2. 初回アクセス時、ブラウザに**参加者ID**が自動発行・保存される（→ §4）。
3. 各ブースのQRを読むたびにスタンプが1つ付与される（同じブースの重複は無効）。
4. スタンプが**規定数**に達すると「ガチャを回す」が解放される。
5. ガチャを回すと抽選で景品が決まる（→ §7）。
6. 景品がデジタルならその場で受け取り、実物なら**引き換え案内**が表示される。

## 4. 参加者の識別（端末識別方式）

受付でのコード配布は行わない。**来場者の端末（ブラウザ）で識別**する。

- 初回アクセス時に一意のID（UUID）を発行し、ブラウザの **localStorage** に保存する。
- 以降のアクセスはそのIDを使ってスタンプ・ガチャの状態を管理する。
- サーバ側（スプレッドシート）にも参加者IDを記録し、スタンプ数・ガチャ履歴を集計する。

### 制約と割り切り

| 事象 | 挙動 | 割り切り |
|---|---|---|
| 別の端末からアクセス | 別人として新IDが発行される | 文化祭の回遊施策としては許容 |
| ブラウザのデータ消去・シークレットモード | 履歴がリセットされ新IDになる | 同上。景品の不正大量獲得は§10で抑制 |
| 1台を複数人で共有 | 同一人物として扱われる | 運用上のレアケースとして許容 |

> 厳密な本人認証は本アプリの目的に対して過剰。手軽さ（インストール・ログイン不要）を優先する設計判断。

## 5. データ設計（スプレッドシートのシート構成）

スプレッドシートをデータベースとして使い、1シート＝1テーブルとする。

### `参加者`
| 列 | 内容 |
|---|---|
| 参加者ID | UUID（端末で発行） |
| 初回アクセス日時 | タイムスタンプ |

### `スタンプ記録`
| 列 | 内容 |
|---|---|
| 参加者ID | 誰が |
| ブースID | どのブースで |
| 取得日時 | いつ |

※ 重複防止キー＝（参加者ID × ブースID）。既に存在する組み合わせは加算しない。

### `景品マスタ`
| 列 | 内容 |
|---|---|
| 景品ID | 識別子 |
| 名称 | 景品名 |
| 種別 | デジタル／実物 |
| 重み | 当選確率の重み付け |
| 在庫 | 残数（実物で使用。デジタルは無制限可） |
| 配信URL／引換場所 | デジタルは配信先、実物は引き換え場所・方法 |

### `ガチャ履歴`
| 列 | 内容 |
|---|---|
| 参加者ID | 誰が |
| 当選景品ID | 何が当たったか |
| 抽選日時 | いつ |

### `引き換え記録`
| 列 | 内容 |
|---|---|
| 引換コード | ワンタイムの一意コード（実物景品の引き換え検証に使用） |
| 参加者ID | 誰が |
| 景品ID | 何を |
| 引換状況 | 未引換／引換済 |
| 引換日時 | 引き換えが完了した日時 |

## 6. 画面構成

UIの設計思想・見た目は前単元「コンテンツ」の成果（[02_コンテンツ/授業/03](../02_コンテンツ/授業/03_コンテンツの制作②_UIとアクセシビリティ.md)）を組み込む。本アプリで用意する画面：

| 画面 | 内容 |
|---|---|
| トップ | 現在のスタンプ数・あと何個でガチャか・ブース一覧の達成状況 |
| ガチャ | 「回す」操作・抽選演出 |
| 結果／引き換え案内 | 当選景品の表示・デジタル受け取り or 実物引き換え案内 |
| エラー | 無効なQR・通信失敗・在庫切れ時のメッセージ |

## 7. 主要処理ロジック

| 処理 | 内容 |
|---|---|
| スタンプ付与 | QRのブースIDを受け取り、`スタンプ記録`に追記。**同一参加者×同一ブースの重複は無効** |
| 達成判定 | 参加者のスタンプ数が規定数以上かを判定し、ガチャ解放可否を返す |
| ガチャ抽選 | `景品マスタ`の重みに基づく重み付き抽選。**在庫0の景品は対象外**。全て在庫切れ時の挙動も定義 |
| 景品引き換え（デジタル） | 当選後すぐに配信URLを表示して受け取らせる（在庫無制限のため即時発行） |
| 景品引き換え（実物） | 一意の**引換コード**を発行し`引き換え記録`に`未引換`で登録。**当選時に在庫を1確保（予約）**し、引き換えは後述のスタッフ検証フローで完了させる（→ §8.5） |

## 8. QRコード設計

- ブースごとにWebアプリのURLへ**ブースIDをパラメータで付与**する（例：`https://<webapp>/exec?booth=A1`）。
- ブースIDで「どのブースのスタンプか」をサーバが判別する。
- QR画像は各ブース分を生成・印刷して掲示する。

## 8.5 実物景品の引き換えフロー

実物景品は在庫が有限のため、**不正なく確実に1回だけ引き換える**ことが必要。検証は「当選画面の見た目」ではなく、**サーバ側の一回限りの状態遷移**で行う。

```
[来場者] 実物景品に当選
   │ サーバが引換コードを発行・在庫を1確保（予約）
   ▼
[来場者スマホ] 引換コードをQRで表示
   │ 引き換え場所で提示
   ▼
[スタッフ専用画面] QRをスキャン
   ├ 未引換 → 景品名を表示 → 景品を渡す →「引換完了」→ 引換済に更新
   └ 引換済 → 「引き換え済み」と拒否
```

- 「引換完了」での `未引換 → 引換済` の更新が**一回限りの状態遷移**。同じコードを再提示しても弾かれるため、スクショの使い回し・友達への共有では二重に受け取れない。
- 在庫は**当選時に確保**しているので、引き換えに来た来場者へ必ず景品を渡せる（在庫より多く当選することがない）。

### 運営専用の引き換えページ

来場者用画面とは独立した**スタッフ専用ページ**を用意する（来場者が自分で`引換済`にできないようにするため必須）。

- **アクセス制限**：来場者ページとは別のモード／URL（例：`?mode=staff`）にし、**パスコード等でゲート**する。
- **画面要素**：
  - 引換QRの読み取り（Chromebookのカメラ＋JSライブラリ）
  - **引換コードの手入力欄**（後述のフォールバック）
  - 照合結果の表示（景品名・引換状況）
  - 「引換完了」ボタン（`引換済`へ更新）

### 読み取り手段とフォールバック

- **想定端末は Chromebook**。GAS WebアプリはHTTPS配信なのでブラウザのカメラAPIでQRスキャンが可能。
- ⚠️ **構築上のリスク**：GASの `HtmlService` は中身を iframe で表示するため、**iframe にカメラ権限が渡らずスキャンが動かない場合がある**。Chromebook実機＋デプロイ済みURLで**早期に動作検証**すること。
- **手入力フォールバック**：カメラが使えない／読み取れない場合に備え、スタッフが**引換コードを直接入力**して照合できるようにする。引換コードは短い英数字とし、スキャンと手入力のどちらでも同じ照合・状態遷移を通す。

> 引き換えの可否判定（未引換のみ可・一回限りの状態遷移）は担当パーツ「引換コード検証関数」が純粋関数として担う。引換QRの生成・スタッフページ・カメラ検証・状態の書き込みは教師スケルトン側で担保する。手入力フォールバックも検証関数は同じ（コードを渡して判定するだけ）。

## 9. 教師スケルトン vs 生徒担当パーツの境界

専門知識が必要な**共通基盤と、スプレッドシート・Webアプリへの入出力（I/O）は教師がスケルトンとして実装済み**で提供する。生徒の担当パーツは、**スケルトンが呼び出す「純粋関数」（素のデータを受け取り、結果を返す関数）だけ**に限定する。これにより生徒は自分のGASコピー／サンドボックスでサンプルデータを使って関数を単体開発・検証でき、統合（マージ）は教師が行う。

### 教師スケルトンが持つもの（I/Oと共通基盤）

| 区分 | 内容 |
|---|---|
| 識別基盤 | 参加者ID発行・localStorage保存・サーバ記録 |
| 記録基盤 | QR読取受付・スタンプ記録などスプレッドシートへの全読み書き |
| Web公開・画面 | GAS Webアプリのデプロイ・URL発行・HTML（画面）描画 |
| スタッフ用引換ページ | アクセス制限（パスコード）・QR読取／手入力の基盤・Chromebookでのカメラ検証・引換QR生成・状態の書き込み |
| 景品データ登録 | 景品マスタへのラインナップ登録（景品の中身は前単元「コンテンツ」の成果） |
| 関数の呼び出し配線 | 下記の生徒担当関数を呼び出し、戻り値を使う |

### 生徒担当パーツ（純粋関数）

いずれも `入力（素のデータ）→ 出力（値）` の純粋関数。スケルトンがデータを渡し、戻り値を使う。

| 担当パーツ（関数） | 入力 → 出力 |
|---|---|
| ガチャ抽選関数 | 景品リスト（重み・在庫）→ 当選景品id |
| スタンプ達成判定関数 | スタンプ記録（ブースidの配列）→ ガチャ可否＋あと何個 |
| 重複/不正チェック関数 | 既存記録＋新規読取 → 受理/拒否＋理由 |
| 運用データ集計関数 | 記録の配列 → 集計結果（時間帯別・ブース別など） |
| 引換コード検証関数 | 引換記録（コード→状態）＋提示コード → 引換可否＋景品＋新状態 |
| 表示メッセージ生成関数 | 状況データ（達成数・当選景品・エラー種別）→ 表示文言（文字列） |

> すべて純粋関数のため、生徒は **関数のひな形＋サンプルデータ＋テスト呼び出しコード** を受け取り、全体構造を理解しなくても自分の担当部分を単体で開発・検証できる。HTMLの画面描画・スプレッドシートI/O・スタッフページ・引換QRはスケルトン側に寄せ、生徒は計算・判定・文言生成のロジックだけを担う。

## 10. 不正・エラー対策の考え方

担当パーツ「重複/不正チェック関数」の検討素材。完璧な防止ではなく、**文化祭として妥当な範囲の抑制**を目指す。

- 同一参加者×同一ブースの**重複スタンプは無効**（§7）。
- 同じブースQRの**連続・短時間の再読み**を弾くルールを検討（生徒が設計）。
- **実物景品のスクショ使い回し・友達への共有**は、引換コードの`未引換 → 引換済`の一回限り遷移（§8.5）で無効化する。
- localStorage消去による**ガチャの引き直し**は完全には防げないが、スタンプの再収集が必要でコストが高く、実物景品は**当選時の在庫確保**で総数が上限を超えないため、大量獲得を抑制できる。

## 11. 運用前提とデータ引き渡し

- **ネットワーク**：来場者のスマホがインターネット接続できること（校内Wi-Fi／モバイル回線）。
- **端末**：来場者の私物スマホが基本。受付・引き換え用に学校タブレットを併用。
- **データ引き渡し**：`スタンプ記録`『ガチャ履歴』『引き換え記録』を、04データサイエンス単元の分析データとして引き渡す。時間帯別アクセス・ブース別人気・ガチャ利用率などが分析テーマになる（担当パーツ「運用データ集計関数」で集計する）。
