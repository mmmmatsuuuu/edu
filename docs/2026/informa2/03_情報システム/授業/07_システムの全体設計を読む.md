# 第7時：システムの全体設計を読む

## 本時の目標

- システム設計の基本的な進め方（要件定義・フローチャート・状態遷移・モジュール分割）を理解する。
- 教師が提供するスケルトン（ひな形）の構造と処理の流れを読み解ける。
- スタンプラリーの全体の中で、自分が担当しうるパーツがどこに入るかを把握し、次時の担当決めに見通しを持てる。

## キーワード

| 用語 | 説明 |
|---|---|
| 要件定義 | システムに「何が必要か・何をするか」を整理して決めること |
| フローチャート | 処理の流れを記号と矢印で表した図 |
| 状態遷移 | 条件によってシステムの状態が移り変わること（例：スタンプ数→ガチャ可否） |
| モジュール | システムを役割ごとに分けた部品。組み合わせて全体を作る |
| スケルトン（ひな形） | 共通部分を作り込み、担当箇所を空けてある土台のプログラム |
| 擬似コード | 実際の文法ではなく、処理の流れを言葉で書き表したもの |

## 準備物・教材

- 教師：授業動画（システム設計の進め方＝要件定義・フローチャート・状態遷移・モジュール分割）、スライド
- 教師：スタンプラリーのスケルトン構造図・処理フロー・担当パーツの位置づけ資料（アプリ設計 [03_アプリ設計.md](../03_アプリ設計.md) の §7 処理ロジック・§8.5 引き換えフロー・§9 境界表を基に作成）
- 教師：担当パーツ一覧（ガチャ抽選／景品・引き換え／スタンプ達成判定／記録・集計／不正・エラー対策／画面文言）
- 生徒：1人1台端末、Google Classroom、（可能なら）スケルトンの閲覧環境
- 教科書：情報Ⅱ（東京書籍）第4章

## 授業展開

| 時間 | 段階 | 学習活動 | 指導上の留意点 |
|---|---|---|---|
| 5分 | 導入 | これまでの座学（通信・データ・QR・コード・AI）を振り返り、「部品がそろった。次は全体をどう組むか」を提示 | 既習がすべて全体設計につながることを意識させる |
| 18分 | 展開①（動画視聴・インプット） | 授業動画を視聴：システム設計の進め方を学ぶ。要件定義（何を作るか整理）→ フローチャート（処理の流れ）→ 状態遷移（状態の移り変わり）→ モジュール分割（役割ごとの部品化）。擬似コードで流れを書く方法も紹介 | 動画はアプリには触れず一般知識として扱う。図（フローチャート・状態遷移図）の読み方を中心に、手順を順序立てて提示する |
| 20分 | 展開②（確認・深化／スケルトン読解） | ①教師提供スケルトンの全体構造（モジュール構成）と処理の流れを、フローチャート・状態遷移（スタンプ数→ガチャ可否→景品引き換え）で読み解く → ②各モジュールが担当パーツ一覧のどれに対応するかを確認し、「専門知識が要る共通部分（教師実装）」と「自分たちが穴埋めするパーツ」の境界を把握する | コードの細部ではなく構造と流れに注目させる。担当パーツがどこに入るかを地図的に示し、次時の希望決めの判断材料にする。少人数を生かし個別に疑問を拾う |
| 7分 | まとめ | キーワード整理。「全体設計の中で自分の担当が一部分として入る」ことを確認。次時（演習①：担当パーツ決め）へ接続し、各自どのパーツに興味があるか軽く考えてくるよう促す | 担当決めは次時に正式に行うことを伝え、難易度・役割（責任者制）の予告をする |

## 板書・スライド構成

1. 本時のテーマ：システムの全体設計を読む
2. 設計の進め方：要件定義 → フローチャート → 状態遷移 → モジュール分割
3. フローチャート・状態遷移図の読み方
4. ＜深化＞スケルトンの全体構造（モジュール構成）
5. 状態遷移：スタンプ数 → ガチャ可否 → 景品引き換え
6. 共通部分（教師実装）と担当パーツ（穴埋め）の境界マップ
7. 担当パーツ一覧と次時の予告（希望決め・責任者制）

## 本時の評価

- **観点：知識・技能**（評価方法：ワークシート・授業内の発言）
  - 要件定義・フローチャート・状態遷移・モジュール分割の意味を説明できているか。
  - スケルトンの処理の流れを読み、担当パーツがどこに入るかを説明できているか。
- 形成的評価とし、演習（担当パーツの設計・実装）と単元テストにつなげる。

## 備考・メモ

- 「スケルトン」は現場で使われる用語のため本単元でもそのまま用いるが、初出時に「ひな形」と補足し意味を取らせる。
- 本時は座学から演習への橋渡し。コードの細部に踏み込まず、構造・流れ・担当の地図づくりに徹する。
- 動画は一般知識（設計手法）に徹し、スケルトンとの関連づけは後半の読解活動で行う。
