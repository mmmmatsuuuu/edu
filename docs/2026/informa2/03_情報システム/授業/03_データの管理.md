# 第3時：データの管理

## 本時の目標

- データを「表（テーブル）」で管理する考え方を理解し、レコード・フィールド・主キーを説明できる。
- テーブル同士をリレーションでつなぐリレーショナルデータベース（RDB）の意味と、正規化によって整合性が保たれることを理解する。
- スプレッドシートを簡易的なデータベースとして使えることを理解し、スタンプラリーのデータをどんなテーブルで持つかを読み解ける。

## キーワード

| 用語 | 説明 |
|---|---|
| データベース | 大量のデータを整理して蓄積し、検索・更新しやすくした仕組み |
| エンティティ（実体） | データとして管理する対象となる「もの・人・出来事」（例：参加者・景品・引き換え） |
| テーブル・レコード・フィールド | データを表で管理する単位。行＝レコード（1件分）、列＝フィールド（項目）。1つのエンティティが1つのテーブルになる |
| 主キー | 各レコードを一意に識別する項目（例：参加者ID） |
| 外部キー | 別テーブルのレコードを参照するための項目（例：引き換え記録の参加者ID・景品ID） |
| リレーショナルデータベース（RDB） | テーブル同士をリレーション（関連）でつないで管理するデータベース。最も広く使われる方式 |
| リレーション（テーブルの関連） | あるテーブルの項目を別テーブルと結びつけること（例：引き換え記録 ↔ 参加者／景品マスタ） |
| 正規化 | データの重複をなくし整合性を保てるよう、テーブルを適切に分けて設計すること |
| ER図（実体関連図） | エンティティ同士の関係を図で表したもの。DB設計の全体像を見る図（本時は存在を知る程度） |
| SQL | データベースからデータを検索・抽出・操作するための言語（本時は概念紹介のみ） |
| 表計算ソフト | スプレッドシートなど表形式でデータを扱うソフト。簡易的なデータベースとしても使える |

## 準備物・教材

- 教師：スタンプラリーのテーブル一覧とエンティティ関連の簡易図（アプリ設計 [03_アプリ設計.md](../03_アプリ設計.md) の §5 データ設計を基に作成）
- 生徒：1人1台端末、Google スプレッドシート、Google Classroom
- 教科書：情報Ⅱ（東京書籍）第4章

## 授業展開

| 時間 | 段階 | 学習活動 | 指導上の留意点 |
|---|---|---|---|
| 5分 | 導入 | 前時の復習（サーバがデータを書き込む）。「もし1枚の表に全部の情報を詰め込んだらどんな困りごとが起きるか」を問いかける | 重複・書き換え漏れなど"整合性"の問題に気づかせ、テーブルを分ける動機づけにする |
| 20分 | 展開①（動画視聴・インプット） | 授業動画を視聴：データベースとテーブル・レコード・フィールド・主キー、RDBとリレーション・外部キー、正規化の意味。**ER図は具体例（例：顧客・注文・商品）で関係を図示しながら**、**SQLは具体例（例：`SELECT 氏名 FROM 顧客 WHERE 年齢 >= 20`）で「何を取り出す指示か」を**紹介 | 動画はアプリには触れず一般知識として扱う。ER図・SQLは用語だけにせず具体例とセットで提示し理解が薄くならないようにする。SQL・ER図は「存在と役割を知る」レベルに留める |

| 7分 | まとめ | キーワード整理。「テーブルを分けてリレーションでつなぐ＝整合性の取れたDB」を確認。スプレッドシートがDB代わりになることを押さえ、次時（QRコードとWebアプリ）へ接続 | スタンプを"書き込む入口"がQRであることを次時の問いとして残す |

## 板書・スライド構成

1. 本時のテーマ：データの管理
2. 1枚の表に詰め込む問題（重複・整合性の崩れ）
3. テーブル・レコード・フィールド・主キー
4. RDBとリレーション／外部キー（一般例：顧客 ↔ 注文）
5. 正規化＝重複をなくし整合性を保つ
6. ER図の例（顧客・注文・商品）／SQLの例（`SELECT … FROM … WHERE …`）
7. ＜深化＞スタンプラリーのテーブル一覧と簡易ER図：引き換え記録（参加者ID・景品ID）↔ 参加者／景品マスタ
8. スプレッドシート＝1シート1テーブル

## 本時の評価

- **観点：知識・技能**（評価方法：ワークシート・授業内の発言）
  - テーブル・レコード・フィールド・主キー／外部キーを説明できているか。
  - テーブルを分けてリレーションでつなぐ理由（整合性）を、スタンプラリーの例で説明できているか。
- 形成的評価とし、単元末の単元テストにつなげる。

## 備考・メモ

- ER図・SQLは「存在と役割を知る」程度に留めるが、用語の丸暗記にならないよう必ず具体例とセットで提示する。
- 前単元評価の「手順に弱い」特性に配慮し、正規化は「1枚表→分割」の手順を実際に動かして体感させる。
- 動画は一般知識に徹し、アプリとの関連づけは後半の確認・深化活動で行う。
