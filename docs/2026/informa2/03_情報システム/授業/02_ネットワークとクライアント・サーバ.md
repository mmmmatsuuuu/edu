# 第2時：ネットワークとクライアント・サーバ

## 本時の目標

- ネットワーク上で「クライアント」と「サーバ」が役割を分担して通信していることを説明できる。
- リクエストとレスポンスの流れ、およびリクエストの種類（GET・POST など）を理解する。
- 「QRを読む → サーバに記録される」という処理を、クライアント・サーバの通信として捉えられる。

## キーワード

| 用語 | 説明 |
|---|---|
| ネットワーク | 複数のコンピュータをつないで情報をやり取りできるようにした仕組み |
| クライアントとサーバ | サービスを要求する側（クライアント）と、要求に応えて提供する側（サーバ）の役割分担 |
| P2P（ピアツーピア） | サーバを介さず、機器同士が対等に直接データをやり取りする方式。クライアント・サーバ型との対比で押さえる |
| TCP/IP | インターネットで通信するための共通ルール（プロトコル）の集まり |
| IPアドレス | ネットワーク上の機器を識別するための番号 |
| HTTP | Webページなどをやり取りするためのWeb通信のルール |
| HTTPメソッド（GET / POST / PUT / DELETE） | リクエストの種類。取得（GET）・送信や作成（POST）・更新（PUT）・削除（DELETE）など「何をしたいか」を表す |
| DNS | ドメイン名（例：example.com）をIPアドレスに変換する仕組み |
| リクエストとレスポンス | クライアントの要求（リクエスト）とサーバの応答（レスポンス）のやり取り |

| 5分 | まとめ | キーワード整理。「スタンプラリーはクライアント・サーバ型の通信である」ことを確認し、次時（データの管理）へ接続 | 「サーバが書き込む“データ”はどんな形でためられるのか」を次時の問いとして残す |

## 本時の評価

## 備考・メモ

- P2Pは本アプリでは使用しないが、クライアント・サーバ型の特徴を際立たせる対比として扱う。
- HTTPメソッドは「GET＝取得／POST＝送信」を中心に、PUT・DELETEは種類の紹介にとどめ、深入りしない。
- 動画は一般知識に徹し、アプリとの関連づけは後半の確認・深化活動で行う。
