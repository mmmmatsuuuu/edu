# 第12時：演習⑤｜結合テストとリハーサル

## 本時の目標

- 全パーツを統合したシステムを、本番を想定してテストできる。
- 見つかった不具合を切り分けて修正できる。
- 景品引き換え（引換QR・スタッフ検証・手入力フォールバック）の運用を練習し、検証記録をまとめられる。

## キーワード

| 用語 | 説明 |
|---|---|
| 結合テスト | 個々のパーツを組み合わせた状態で、全体が正しく動くか確かめるテスト |
| テストケース | 「この入力ならこの結果」という確認項目。本番で起こりうる場面を想定して用意する |
| 検証記録 | テスト項目・見つけた不具合・改善依頼をまとめた記録。検証責任者がまとめる |
| 引き換えフロー | 実物景品を引換QR＋スタッフ確認で一回限り引き換える流れ（手入力フォールバックを含む） |

## 準備物・教材

- 教師：統合済みシステム、テストケース一覧のひな形、検証記録テンプレート、スタッフ用引き換えページ
- 教師：Chromebook（引換QRの読み取り検証用）、引換コードの手入力フォールバック手順（アプリ設計 [03_アプリ設計.md](../03_アプリ設計.md) の §8.5 を参照）
- 生徒：1人1台端末、各自の担当パーツ、Google Classroom
- 教科書：情報Ⅱ（東京書籍）第4章

## 授業展開

| 時間 | 段階 | 学習活動 | 指導上の留意点 |
|---|---|---|---|
| 5分 | 導入 | 提出された各関数を**教師がスケルトンへ統合した結合版**を確認し、本時で「通しで動くか」を検証することを共有 | 自分の関数だけでなく全体の動作に視野を広げさせる |
| 15分 | 活動①（結合テスト） | スタンプ→ガチャ→景品引き換えの一連を、テストケースに沿って通しで確認する。重複スタンプ・在庫切れ・達成前のガチャなど想定場面も試す | 「正常系」だけでなく「異常系」も試させる。不具合はどのパーツ由来か切り分ける |
| 13分 | 活動②（引き換えリハーサル） | 実物景品の引き換えを練習：来場者が引換QRを提示 → スタッフがChromebookで読み取り → 引換済に更新。**カメラが動くかを実機で検証**し、読めない場合は**手入力フォールバック**を試す | カメラのiframe問題が起き得るため必ず実機検証。手入力でも同じ結果になることを確認させる |
| 10分 | 活動③（修正・検証記録） | 見つかった不具合を修正し、検証責任者を中心に検証記録（テスト項目・不具合・改善依頼）をまとめる | 直しきれない点は「本番での運用回避策」も検討させる |
| 7分 | まとめ | 本番に向けた残課題・役割分担（運用当日の動き）を確認 | 当日のトラブル対応の心構えを共有し、不安を下げる |

## 板書・スライド構成

1. 本時のテーマ：結合テストとリハーサル
2. 結合テスト：通しで動かす（正常系／異常系）
3. テストケース例（重複・在庫切れ・達成前ガチャ）
4. 引き換えリハーサル：引換QR → スタッフ確認 → 引換済
5. カメラ検証と手入力フォールバック
6. 検証記録のまとめ／当日の役割分担

## 本時の評価

- **観点：思考・判断・表現**（評価方法：検証記録）
  - テストで不具合を見つけ、原因を切り分けて改善策を示せているか。
- **観点：主体的に学習に取り組む態度**（評価方法：活動状況）
  - 本番を想定し、チームで粘り強くテスト・修正に取り組んでいるか。
- 検証責任者の記録を中心に評価する。

## 備考・メモ

- 引換QRのカメラ読み取りは本番前にChromebook実機で必ず確認し、手入力フォールバックの運用も決めておく。
- 当日の役割（運用係・引き換え係・トラブル対応係）を本時の終わりに割り振っておくと当日が円滑。
