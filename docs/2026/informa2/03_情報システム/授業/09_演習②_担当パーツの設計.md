# 第9時：演習②｜担当パーツの設計

## 本時の目標

- 担当パーツの要件を整理し、仕様（何をどう動かすか）を具体的に決められる。
- 設計の意図や、選ばなかった案とのトレードオフを言語化できる。
- 設計責任者を中心に、チームで設計記録をまとめられる。

## キーワード

| 用語 | 説明 |
|---|---|
| 仕様 | パーツが「何を・どう動かすか」を具体的に決めた取り決め |
| トレードオフ | あちらを立てればこちらが立たない関係。複数案から選ぶときの判断材料 |
| 設計記録 | 仕様・設計意図・迷った選択肢をまとめた記録。設計責任者がまとめ、評価のエビデンスになる |
| 擬似コード | 実際の文法ではなく、処理の流れを言葉で書き表したもの |

## 準備物・教材

- 教師：設計ワークシート（要件・仕様・トレードオフ・擬似コード/フローチャートの記入欄）、設計記録テンプレート
- 教師：関数ごとの設計の論点例と入出力の仕様（例：ガチャ抽選＝重み・在庫切れ時の戻り値、達成判定＝必要スタンプ数・重複の数え方、引換コード検証＝未引換のみ可。アプリ設計 [03_アプリ設計.md](../03_アプリ設計.md) の §9 の入出力表を参照）
- 生徒：1人1台端末、Google Classroom
- 教科書：情報Ⅱ（東京書籍）第4章

## 授業展開

| 時間 | 段階 | 学習活動 | 指導上の留意点 |
|---|---|---|---|
| 5分 | 導入 | 設計フェーズの位置づけ（実装の前に何を決めるか）を確認し、設計責任者の役割を共有 | 「作業は全員で・記録は責任者」を再確認 |
| 15分 | 活動①（要件整理・仕様決め） | 担当する関数の**入力（受け取るデータ）と出力（返す値）**を確認し、仕様を具体的に決める（例：達成スタンプ数、ガチャの当選確率と在庫切れ時の戻り値、引換コード検証の可否ルール） | 入力→出力の形を最初に固定すると指示・検証がぶれないことを伝える。論点例を示し決めるべき項目を漏らさないよう支援する |
| 15分 | 活動②（トレードオフの検討・図化） | 仕様の選択肢を比較し、なぜその案にしたか・どの案と迷ったかを検討。処理の流れをフローチャート／擬似コードで表す | 「概念・記述が得意」な強みを生かし、判断の理由を言語化させる。手順は図で順序立てて表現させる |
| 10分 | 活動③（設計記録の作成） | 設計責任者を中心に設計記録をまとめる。**各自が自分の視点で1項目は記述**し、全員の思考が残るようにする | 設計記録が観点別評価（思考・判断・表現）のエビデンスになることを意識づける |
| 5分 | まとめ | 設計の要点を共有し、次時（AIで実装）の準備（プロンプトに何を伝えるか）を確認 | 設計が決まっていれば実装の指示がぶれないことを伝える |

## 板書・スライド構成

1. 本時のテーマ：担当パーツの設計
2. 設計フェーズで決めること（要件 → 仕様）
3. パーツ別の論点例（確率・在庫・達成数・引き換え方法）
4. トレードオフ：選んだ案／迷った案と理由
5. フローチャート・擬似コードで流れを表す
6. 設計記録のまとめ方（各自1項目）

## 本時の評価

- **観点：思考・判断・表現**（評価方法：設計記録・ワークシート）
  - 要件を整理して仕様を決められているか。
  - 設計意図やトレードオフを自分の言葉で説明できているか。
- 設計責任者の記録を中心に、各自の記述も評価対象とする。

## 備考・メモ

- 設計はこの後の実装の指示書になる。仕様が曖昧だとAIへの指示もぶれるため、具体化を重視する。
- 引換コード検証関数は「未引換のみ可・引換済/不存在は拒否」という一回限りの判定ルールを設計に含める（引換QR・スタッフページ・在庫の書き込みはスケルトン側）。
